## 多引擎 AI 3D 生成架构开发

### 需求背景
用户要求增加腾讯混元3D的API接口，使项目可以同时测试混元3D和Tripo3D两个AI引擎。

### 混元3D API 调研结果
- **产品名称**: 腾讯混元生3D（独立产品，product ID 1804）
- **请求域名**: `ai3d.tencentcloudapi.com`
- **认证方式**: TC3-HMAC-SHA256 签名（腾讯云 API 3.0）
- **API版本**: `2025-05-13`
- **核心接口**:
  - `SubmitHunyuanTo3DProJob` — 提交3D专业版生成任务
    - 输入: ImageUrl/ImageBase64/Prompt, EnablePBR, FaceCount, GenerateType
    - 输出: JobId
  - `QueryHunyuanTo3DProJob` — 查询任务状态
    - 输入: JobId
    - 输出: Status(WAIT/RUN/FAIL/DONE), ResultFile3Ds[{Url, PreviewImageUrl, Type}]
- **其他接口**: SubmitHunyuanTo3DRapidJob(极速版), SubmitProfileTo3DJob(人物3D), SubmitTextureTo3DJob(纹理), SubmitReduceFaceJob(智能拓扑), Convert3DFormat(格式转换)

## 生成失败后自动跳回登录页修复 — 全链路（研发-B）

### 问题
AI 生成失败后展示了失败页面，但随即自动跳转回登录页。第一轮仅修复轮询 `silent401` 后问题依旧。

### 根因（深入排查 6 个缺口）
1. 轮询请求（已修复）：`checkStatus()` 的 `GET /model/:id/status` 已有 `silent401: true`
2. **生成请求缺口**：`startGenerate()` 的 `POST /model/generate` 没有 `silent401`，401 → 清 token + setTimeout(reLaunch, 1.5s)，即使 catch 中 `redirectTo` 到失败页，1.5s 后仍会跳登录页
3. **重试请求缺口**：`doRetry()` 的 `POST /model/:id/retry` 同理
4. **上传请求缺口**：`wx.uploadFile` 不走 request.ts，401 未被检测
5. **裁剪接口缺口**：`post('/photo')` 和 `put('/photo/:id/crop')` 未加 silent401
6. **setTimeout 不可取消 + 无防重入**：多个请求同时 401 会重复清 token 和重复跳转

### 修复（全链路）
- `generate-waiting.ts`：`startGenerate()` 加 `{ silent401: true }`，catch 中自行处理 TOKEN_EXPIRED
- `generate-failed.ts`：`doRetry()` 加 `{ silent401: true }`，catch 中处理 TOKEN_EXPIRED
- `photo-crop.ts`：`uploadToServer()` 增加 401 检测返回 TOKEN_EXPIRED；`post`/`put` 加 `{ silent401: true }`；`onConfirm()` catch 处理 TOKEN_EXPIRED
- `request.ts`：新增全局 `isHandling401` 防重入标记；非静默 401 的 reject 也统一用 `TOKEN_EXPIRED`

## model 路由 koa-router 中间件 Bug — 真正根因（研发-A）

### 问题
`POST /api/model/generate` 和 `POST /api/model/:id/retry` 始终返回 401，即使 token 完全有效。刚登录获取的 token 立即请求就 401。同一个 token 请求 GET 接口（model/list、user/info）全部正常。

### 根因
koa-router 的 `router.use(authMiddleware)` 在 `router.get('/engines', ...)` 之后注册时，对后续 **POST** 路由的中间件应用存在兼容性问题。所有 model GET 路由正常通过鉴权，但所有 model POST 路由被绕过鉴权直接返回 401。其他路由模块（photo、order）不受影响，因为它们没有在 `router.use()` 之前注册免鉴权路由。

### 修复
将 `router.use(authMiddleware)` 改为每个路由**单独内联** `authMiddleware`：
```ts
router.post('/generate', authMiddleware, async (ctx) => { ... });
router.get('/:id/status', authMiddleware, async (ctx) => { ... });
```
修改文件：`server/src/routes/model.ts`

## auth.ts `await next()` 位置导致假 401 修复（研发-A）

### 问题
即使 model 路由已改为内联 authMiddleware，`POST /api/model/generate` 仍然返回 401。

### 根因
`server/src/middlewares/auth.ts` 中 `await next()` 放在 JWT 验证的 `try-catch` 块内。后续路由处理器（如 `prisma.dailyUsage.upsert()`）抛出的数据库异常被 auth 的 catch 捕获，错误地返回了 401 状态码。

### 修复
将 `await next()` 从 JWT try-catch 中移出。JWT 验证失败走 catch + return；通过后执行 `await next()`，后续路由异常正确冒泡到全局 error middleware。

### 验证
`node verify.js` — model/generate 不再返回 401（改为 500 Prisma 错误）

## dailyUsage.upsert() 唯一约束冲突修复（研发-A）

### 问题
auth.ts 修复后，`POST /api/model/generate` 返回 500：`Unique constraint failed on daily_usage_user_id_usage_date_key`

### 根因
`prisma.dailyUsage.upsert()` + MySQL `DATE` 类型 + JS Date 时区偏差：
- `new Date().setHours(0,0,0,0)` 得到本地时区（UTC+8）午夜
- Prisma 传给 MySQL 时转为 UTC（前一天 16:00）
- upsert 的 where 查不到已有记录 → 走 create → 唯一约束冲突

### 修复
1. 新增 `getTodayDate()` — 用 `YYYY-MM-DDT00:00:00.000Z` 构造 UTC 零点日期
2. `consumeUsage()` 从 upsert 改为先查后改（findUnique → update/create）
3. create 外层 try-catch 捕获 P2002 并发竞争 → fallback 到 update
4. `getDailyUsage()` / `refundUsage()` 同步使用 `getTodayDate()`
